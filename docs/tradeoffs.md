# Design Decisions and Tradeoffs

## Per-config QA generation (fairness constraint)

Each chunking configuration generates its own synthetic QA dataset using the chunks produced by that config. The alternative — one shared QA set evaluated against all configs — would bias results toward the config whose chunk boundaries most closely match the QA generation config. Per-config generation means each config is evaluated against ground truth anchored to its own chunk IDs. The tradeoff: QA datasets are not directly comparable across configs; only the IR metrics derived from them are.

## OpenAI text-embedding-3 over SentenceTransformers

P3 uses OpenAI's `text-embedding-3-small` (1536-dim) and `text-embedding-3-large` (3072-dim) rather than local SentenceTransformers models. OpenAI embeddings are state-of-the-art on MTEB and consistent across machines — there's no local GPU requirement, no model download variability, and no tokenizer version mismatch. The tradeoff: API cost (~$0.002/1M tokens for small, ~$0.13/1M for large) and an internet dependency. For a single-PDF grid search with ~1,000 chunks, the cost is negligible. P4 uses local SentenceTransformers to demonstrate the tradeoff explicitly.

## pdfplumber as the only parser (not compared)

The spec allows comparing multiple PDF parsers (pdfplumber, PyPDF2, PyMuPDF). P3 standardises on pdfplumber for all runs. Parser choice is recorded in `ChunkConfig.parser` to prevent accidental cross-parser comparisons. The alternative — a parser grid — would multiply the experiment count by 3 without adding signal for the core question (chunking × embedding × retrieval). Parser comparison is a separate experiment, not this grid.

## Synthetic QA via LLM, hand-authored chunk IDs

QA pairs are generated by prompting an LLM with each chunk's content and asking it to produce a question whose answer is in that chunk. The ground truth is the chunk ID used to generate the question. This is a weaker ground truth than human annotation — the LLM occasionally generates questions that are answerable by adjacent chunks, inflating recall scores. The known limitation is documented in the README. The alternative (human annotation) is not feasible for a 24-config grid.

## MRR and Recall@K as primary metrics (not Precision@K)

With exactly one ground-truth chunk per query, `Precision@K` is capped at `1/K` (0.20 at K=5). It's included in result files for completeness but excluded from the primary comparison table. MRR captures where the first correct result appears; Recall@K captures whether any correct result appeared. Both are meaningful in this single-relevant-chunk regime.

## fixed_256 reference config gap (MRR 0.507 vs spec 0.963)

The spec document uses `fixed_256` as a "before optimization" baseline with reported MRR=0.963. The actual P3 result for `fixed_256_ol50__small__vector` is MRR=0.507. The gap has not been investigated but likely reflects a difference in PDF, QA generation prompt, or overlap parameter between the spec author's run and this implementation. This is documented as an open item — not a silent discrepancy.

## QA dataset cache invalidation strategy

Each chunking config's QA dataset is cached to `data/qa_datasets/{chunk_config_label}.json`. The cache key is the `chunk_config_label` string (e.g. `fixed_256_ol50`). On load, the cached dataset is validated: all chunk UUIDs referenced by cached QA pairs must exist in the current chunk set. If any UUID is missing — which happens when the chunking parameters change and chunks are re-generated with new UUIDs — the cache is silently discarded and the QA dataset is regenerated.

**What invalidates the cache:** changing any parameter that affects chunk boundaries (chunk_size, overlap, breakpoint_threshold, max_sentences, sentences_per_chunk). These produce new chunk UUIDs, which fail the subset check.

**What does NOT invalidate the cache:** changing the QA generation prompt or model. If `_SYSTEM_PROMPT` in `qa_generator.py` is updated to produce different question styles, the cached dataset is returned unchanged because the chunk UUIDs still match. To force regeneration after a prompt change, delete the relevant file in `data/qa_datasets/` or run with `--force` (which bypasses the experiment-level cache but not the QA cache — delete QA files manually). This is a known limitation documented in `docs/failures.md`.

## Resume/skip logic based on experiment_id file existence

A completed experiment writes its result to `experiments/{experiment_id}.json`. On re-run, if that file exists the cell is skipped. This means changing any config parameter changes the experiment_id and forces a fresh run even if only an unrelated parameter changed. The alternative — content-addressed caching — requires hashing the full config, which is harder to inspect and debug. File-existence skipping is transparent: you can see exactly which cells have results.
